## Session: 2026-05-04 — Downlink QoS Debug (完成)

### Bugs Fixed

**1. `_agent_ips` race condition → downlink client 被 skip**

- **症狀**：VO downlink 固定 error，BE 有時正常。Agent 的 iperf3 server 等了 62 秒（= duration + proc.wait timeout）沒有 client 連進來。
- **根因**：Orchestrator 每次 run 都新建一個 instance，`_agent_ips` 靠 heartbeat（QoS 0）填充。如果 heartbeat 在 6.5s 的 grace period 內沒到，`_start_iperf3_downlink_clients` 就 skip 那台 agent。
- **Fix**：Agent 在 ACK payload 裡加入 `"ip"` 欄位（QoS 1，保證到達）；Orchestrator 的 `_on_ack` 同時更新 `_agent_ips`。這樣只要 ACK 到，IP 就確定有了。
- **Files**：`agent/atf_agent/main.py`（_handle_prepare ACK），`controller/atf_ctrl/orchestrator.py`（_on_ack）

**2. sync-agents.py REMOTE_DIR 錯誤**

- **症狀**：新 code sync 後 agent 行為沒變，新加的 log 也沒出現。
- **根因**：`REMOTE_DIR = "/home/mars/syncbench"` 但 systemd service 的 `WorkingDirectory=/home/mars/atf-validator`，sync 到的是死路。
- **Fix**：`REMOTE_DIR = "/home/mars/atf-validator"`

**3. `_run_iperf3_thread` 無 exception guard**

- **症狀**：若 `run_server()` 拋出 exception（如 `subprocess.TimeoutExpired`），thread silent die，agent 不發 result，orchestrator 等到 RESULT_TIMEOUT 才 timeout。
- **Fix**：加 `try/except Exception`，catch 後發 `status="error"` result，確保 orchestrator 能收到。

### QoS Downlink Findings（重要研究發現）

測試環境：Mac mini（Ethernet, 192.168.1.117）→ AP → RPi（WiFi, 802.11ac VHT80, 433 Mbps link rate）

#### 單條測試
| 設定 | 吞吐量 | Retransmits |
|---|---|---|
| Single BE (0x00) | 266 Mbps | 0 |
| Single VO (0xb8) | 80 Mbps | 0 |
| Single VI (0x68) | ~195 Mbps（推估） | 0 |

#### 並行測試（最關鍵）
| 設定 | rpi-sta-01 | rpi-sta-02 | Total | Retrans |
|---|---|---|---|---|
| BE + BE | 142 Mbps | 140 Mbps | 282 Mbps | 0 |
| BE + VO | 13 Mbps | 4.7 Mbps | **17.7 Mbps** ← 崩潰 | 0 |
| BE + VI | 44.9 Mbps | **195.3 Mbps** | 240 Mbps | 0 |
| VO + VO | 45.7 Mbps | 2.7 Mbps | 48.4 Mbps | **115 + 30** |

#### 根本原因分析

**VO (DSCP EF 0xb8) 的問題：AP AC_VO queue overflow**

AP 的 AC_VO queue 是為 voice call 設計（假設每流 64 Kbps - 1 Mbps，小封包）。用 bulk TCP（1500B MTU，80+ Mbps）打進 AC_VO，queue 立刻溢出：
- Single VO：queue 勉強 hold，80 Mbps
- 2x VO：queue overflow → 大量 drop → TCP retransmit → congestion collapse → 45 + 2.7 Mbps
- 1 BE + 1 VO：VO queue overflow 的 retransmit 佔用 airtime → BE 也從 142 掉到 13 Mbps

**VI (DSCP AF31 0x68) 為什麼正常：AP AC_VI queue 是為 video 設計的**

AC_VI queue 預期高 bitrate（10-50 Mbps video），buffer 更大，不會 overflow。
且 AP 的 downlink EDCA 讓 VI 優先於 BE：
- VI: 195 Mbps，BE: 44.9 Mbps（VI 拿到 4x 以上的 airtime）
- **這是 AP downlink QoS scheduling 真正在作用的直接證據。**

#### 結論與建議

1. **VO (0xb8) 不適合 bulk throughput 測試**。它測的是 AC_VO queue overflow behavior，不是 airtime scheduling。
2. **VI (0x68) 是有意義的 QoS 測試 AC class**。能觀察到真實的 AP 下行優先調度。
3. **DSCP EF → TID 6 mapping 在 AP 上是生效的**（否則 VO 應該跟 BE 一樣），但 AC_VO 設計限制讓它不適合 high-throughput 場景。
4. 若要測試 VO 的 airtime fairness，應使用 **UDP + 固定 bitrate**（模擬真實 voice），而非 TCP bulk。
